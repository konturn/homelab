## Sub-Agent Failure Reflection

Noah asked why sub-agents keep floundering on MR work. My analysis:

1. **Context loss is brutal** — sub-agent doesn't know infrastructure quirks
2. **Wrong abstraction level** — treating "create and land MR" as atomic when it's many judgment calls
3. **Skill is happy-path script** — breaks on edge cases, I keep patching
4. **No escalation** — sub-agent spins or gives up instead of asking for help

**Proposed fixes:**
1. Pre-flight validation (check runners, branches, API access)
2. Shrink scope: split into `gitlab-mr-create` and `gitlab-mr-respond`
3. Add escalation triggers (stuck >2 min, unknown errors → notify main, STOP)
4. Pass infrastructure context in spawn call

**Deeper lesson:** I've been trying to be a manager who delegates everything. More effective as a hands-on lead who delegates grunt work while keeping complex judgment calls.
